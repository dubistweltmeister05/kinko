I get asked about this a LOT, so here's another attempt at answering this. I have a small list of things that I consider to be overlapping for these two industries, so here goes nothing - 
1. C  
What else did you expect? This is the bread and butter of everything that exists, lol. Nothing reaches production without some C involved. So, an extensive understanding of its fundamentals is paramount. Go as deep as you can - then go deeper, because there’s always something new to learn. That said, higher-level languages like C++ and Python are also useful.  
  
2. Assembly  
Don’t scroll past this - I’m not insane. Understanding what assembly is and how it works is crucial when debugging embedded code. The code you write runs at the heart of a processor; understanding opcodes, syscalls, and instructions goes a long way. You don’t have to write assembly (hopefully), but you should be able to read it.  
  
3. Computer Architecture  
Probably the most versatile topic you’ll study. RISC-V, x86, ARM - the list goes on. You could literally do a PhD in this. A good grasp of architecture makes you incredibly versatile. If you understand the arch, you can fit into almost any role - IoT, robotics, automotive, industrial systems, you name it.  
  
4. Analog & Digital Electronics  
My dear CS/IT folks - you need to understand what you’re programming for. Transistors, MOSFETs, BJTs, logic gates, mux/demux, timers, counters - these are the fundamentals of hardware. A basic understanding of these will take you a long way in embedded systems.  
  
5. Protocols  
Communication protocols are what make devices talk. Serial, parallel, industrial, cloud, peer-to-peer - these are essential in making systems scale from one device to thousands. Learn a few of these, you’ll thank yourself later.  

6. Debugging Tools
Writing code is one thing. Figuring out why it doesn't work is another.

Unlike traditional software development, embedded engineers often debug both hardware and software simultaneously. A bug could be caused by faulty firmware, a wiring issue, signal integrity problems, incorrect voltage levels, timing violations, or a combination of all of them.

This is where debugging tools become essential.

Learn how to use oscilloscopes, logic analyzers, multimeters, JTAG/SWD debuggers, protocol analyzers, and serial consoles. Learn how to read datasheets, inspect signals, decode communication protocols, and trace faults methodically.

A surprising amount of embedded development is not spent writing code—it's spent figuring out why something that should work doesn't.

The faster you can isolate problems, the more effective you'll become as an engineer.

7. Linux/RTOS  
You thought I forgot this? Once a processor becomes powerful enough, it needs an OS. This is where Linux shines. Believe me - there’s no IoT network today that functions usefully without Linux at some level. So get your hands dirty with it.  
  
8. OOP  
IoT demands abstraction and compartmentalization - the user should not directly access the hardware. That’s where OOP comes in. If you already understand that, you’re good. If not - well, time to learn what OOP stands for.
